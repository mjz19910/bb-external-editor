# What you should add next (highest value first)

## 1) A **central scheduler / orchestrator**

You already have pieces like:

* `farm_manager.ts`
* `manage_empire.ts`
* `jobs_report.ts`
* `retarget_farm.ts`

…but I’d bet they’re still somewhat **script-driven** rather than **state-driven**.

### Add:

`./orchestrator.ts`

### What it should do:

* Maintain a global view of:

  * all rooted servers
  * available RAM per worker
  * active jobs
  * best targets
  * prep state per target
* Decide whether each target should be:

  * **prep**
  * **batch**
  * **cooldown**
  * **ignore**
* Assign jobs based on:

  * RAM efficiency
  * hack level
  * current security/money state
  * profitability

### Why this matters:

Right now your scripts probably “do things.”
The orchestrator should instead answer:

> “Given the entire network right now, what is the most profitable use of every GB of RAM?”

That’s the leap from “automation” to “engine.”

---

## 2) A **true HWGW batch engine**

I see:

* `hack.ts`
* `grow.ts`
* `weaken.ts`
* `prep.ts`

That’s good. But if you haven’t already, you want a proper:

### Add:

* `./batch_manager.ts`
* `./lib/batch.ts`
* `./lib/timing.ts`

### Goal:

Run **synchronized Hack → Weaken → Grow → Weaken** batches with precise offsets.

### Core features:

* compute thread counts
* compute exact launch delays
* prevent overlap collisions
* reserve RAM before dispatch
* handle drift / timing failures
* auto-fallback to prep mode if target drifts

### Why this is huge:

Once your servers are mostly prepped, **batching is where the money explodes**.

If you’re not batching yet, this is **the** next feature.

## 3) A **persistent state system**

You already have:

* `./lib/db/network_map.json`

Good start. But you likely want more than a network map.

### Add:

* `./lib/db/state.json`
* `./lib/state.ts`

### Track things like:

* target health:

  * money %

  * RAM

### Why:

A lot of Bitburner scripts become fragile because they constantly recompute everything from scratch and forget context.

State lets your system become **stable** instead of merely **reactive**.

## 4) A **RAM reservation / resource allocator**

This is one of the most underrated upgrades.

### Add:

`./lib/ram_allocator.ts`

### Problem it solves:

Without centralized RAM accounting, your scripts will:

* overbook servers
* collide with each other
* underuse free RAM
* randomly fail launches

### It should:

* reserve RAM before a job launches
* release RAM on completion
* pack jobs efficiently across hosts
* support priorities:

  * critical weaken > prep grow > hack > optional utility

### Nice extra:

Add **fragmentation-aware packing**:

* put large jobs on big servers
* use leftovers for small utility jobs

This alone can noticeably increase income.

It also helps tune your batch parameters.

## 7) A **target scoring model that’s actually smart**

You’ve already got:

* `best_target.ts`
* `choose_best_target.ts`
* `money_sort_target.ts`
* `min_sec_target.ts`
* `hack_lvl_target.ts`

That’s good, but these sound like **heuristics**, not a unified scoring model.

### Add:

`./lib/score_target.ts`

### Instead of:

* “highest money”
* “lowest security”
* “lowest hack level”

### Use:

A weighted score like:

```ts
score =
  (expectedMoneyPerSec / ramCost)
  * successChance
  * prepReadiness
  * stabilityFactor
```

### Include:

* weaken time
* grow cost
* max money
* min security
* current state
* chance to hack
* available RAM in your fleet

### Why:

The “best target” changes based on your **current infrastructure**, not just server stats.

That’s where most scripts stay too simplistic.

# Important note before continuing

Your current repo has a likely naming mismatch:

You showed files like:

* `./lib/hack.ts`
* `./lib/grow.ts`
* `./lib/weaken.ts`

but the test uses:

```ts
ns.getScriptRam("/lib/weaken.js")
```

If your actual runtime files are **`.ts`**, use:

```ts
ns.getScriptRam("/lib/weaken.ts")
```

Same for hack/grow.

Bitburner is annoyingly literal about this.
